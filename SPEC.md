# SPEC — trakt_folder_sync

Спецификация поведения плагина. Пишется до кода. Язык — человеческий, не технический.

## Философия

Lampa — клиент Trakt. **Trakt — единственный источник правды.** Любое действие пользователя в Lampa транслируется в вызов Trakt API; Lampa никогда не хранит собственного состояния, которое может разойтись с Trakt. На каждой синхронизации Lampa зеркалит то, что сейчас в Trakt.

Последствие: петель «Lampa думает одно, Trakt другое» в принципе не возникает. Если Trakt отдал состояние — Lampa его показала, и точка.

## Категории Lampa

В Lampa два типа папок в Избранном:

- **Теги** — несколько одновременно: Закладки (book), Нравится, Позже, История просмотров.
- **Статусы** — только один: Смотрю (look), Просмотрено (viewed), Запланировано, Продолжение следует (continued), Брошено (thrown).

Плагин работает только с: **book, look, viewed, continued**.

## Маппинг Trakt → Lampa

| Lampa | Trakt | Направление |
|---|---|---|
| Закладки (book) | Watchlist (фильмы + сериалы) | Двусторонняя |
| Смотрю (look) | Сериалы с `completed > 0 && aired > completed` | Trakt → Lampa |
| Просмотрено (viewed) | Фильм в истории / сериал полностью досмотрен и закрыт | Trakt → Lampa (производная) |
| Продолжение следует (continued) | Сериал догнан, но ещё выходит | Trakt → Lampa (производная) |

## Логика статусов сериала

Берём из Trakt:
- `show.status` (поле объекта шоу при `extended=full`) — одно из: `returning series`, `in production`, `planned`, `canceled`, `ended`.
- `progress.next_episode` — следующая серия, которую пользователю осталось посмотреть. Если `null` — пользователь догнан (это ровно тот же сигнал, по которому Trakt UI кладёт / не кладёт сериал в раздел Continue Watching).
- `progress.completed` — количество просмотренных (используется только чтобы отличить «смотрел хоть что-то» от «не смотрел ничего»).

**Решающий сигнал — `progress.next_episode == null`** («догнан»), а НЕ сравнение количества вышло/просмотрено и НЕ проверка `last_episode`.

Почему не `last_episode`: в эндпоинте `/shows/:id/progress/watched` поле `last_episode` — это последняя **просмотренная пользователем** серия, а не последняя вышедшая. Проверять её «просмотренность» бессмысленно — она по определению просмотрена.

Почему не сравнение количеств: если пользователь пропустил серию в середине, но добрал последнюю вышедшую, `aired > completed`, но смотреть прямо сейчас нечего. Trakt это понимает сам: `next_episode` в таком случае `null`.

Правила:

```
completed == 0
  → сериал не смотрели. В look/viewed/continued не попадает.
  → если в watchlist — остаётся только в Закладках.

completed > 0 && next_episode != null
  → есть что смотреть прямо сейчас → Смотрю.

completed > 0 && next_episode == null && status ∈ {ended, canceled}
  → сериал завершён и догнан → Просмотрено.

completed > 0 && next_episode == null && status ∈ {returning series, in production, planned}
  → пользователь догнал, ждёт новые серии → Продолжение следует.
```

Важно: «completed > 0» означает, что **хотя бы один эпизод отмечен в Trakt как просмотренный**. В Trakt это происходит при scrobble'е при воспроизведении либо при ручной отметке через API. Простое открытие карточки в Lampa это не триггерит.

Источник данных: `/sync/watched/shows?extended=full` даёт список сериалов с `show.status`; на каждый — отдельный запрос `/shows/:trakt_id/progress/watched` за `next_episode`. По сути это бесплатная реализация функциональности VIP-эндпоинта `up_next_nitro`: там Trakt отдаёт весь список Continue Watching разом, мы получаем тот же результат через 1+N запросов. Trakt rate limit 1000/5мин выдерживает десятки запросов на синхронизацию с запасом.

## Логика статусов фильма

Фильм попадает в **Просмотрено**, если он есть в истории Trakt (`/sync/history`, либо `plays > 0` у фильма в watched-списке).

Фильм в **Смотрю/Продолжение следует** не попадает никогда — эти статусы только для сериалов.

## Ручные действия пользователя

Принцип: **действие пользователя в Lampa = вызов Trakt API**. Lampa свой статус не меняет до подтверждения от Trakt (оптимистично отображает, но источник истины — Trakt после следующего чтения).

| Действие в Lampa | Вызов Trakt API |
|---|---|
| Добавить в Закладки | `POST /sync/watchlist` |
| Убрать из Закладок | `POST /sync/watchlist/remove` |
| Фильм → Просмотрено | `POST /sync/history` (добавить в историю) |
| Сериал → Просмотрено | `POST /sync/history` на все вышедшие эпизоды (перечень берём из `progress.seasons[].episodes[]`, шлём одним запросом). После этого Trakt отдаст сериал либо как Просмотрено (если ended/canceled), либо как Продолжение следует (если ещё выходит). Пользователь увидит конечный статус после следующей синхронизации. |
| Убрать из Просмотрено — фильм | `POST /sync/history/remove` с `movies`. |
| Убрать из Просмотрено — сериал | `POST /sync/history/remove` на все вышедшие эпизоды (тот же список, что при добавлении). |
| Сериал → Смотрю | `POST /sync/history` на `S01E01` (минимальная «подсказка Trakt'у»: completed становится > 0, next_episode указывает на S01E02, классификатор даёт look). Остальные эпизоды не трогаем. |
| Убрать из Смотрю | **No-op.** look — производное состояние, нет обратного действия, которое мы могли бы однозначно выразить в Trakt'е. На следующей синхронизации карточка вернётся туда, где ей место по данным Trakt. |
| Фильм → Смотрю, любое → Продолжение следует | **No-op.** Фильм в Смотрю не попадает в принципе. Продолжение следует — автоматическая производная «догнан + выходит», её пользователь не ставит. |

Ключевое: пользователь не «переносит» сериал в Просмотрено — он создаёт **условия**, при которых Trakt отдаст правильный статус. Если сериал ещё выходит, он окажется в Продолжение следует, а не в Просмотрено — это корректно.

Pending Ops для статусных папок (см. §«Задержки Trakt API...»): после успешной записи кладём op `{folder, action, id, type}`. На ближайших синхронизациях (пока op не протух) карточка не будет «уехать» из целевой папки из-за устаревших данных Trakt. Семантика: `add` в папку X → принудительно ТОЛЬКО в X (чистим id из остальных статусных desired); `remove` → чистим id из всех статусных desired.

**Определение типа карточки (show/movie) — через Trakt, не по полям карточки.** В листенере `Favorite.listener` Lampa регулярно отдаёт «тощие» карточки без `method`/`card_type`/`media_type`/`number_of_seasons` — проверять их эвристикой бессмысленно, отрицательный ответ «нет признаков сериала» ≠ «это фильм» (такой же баг был с book в 0.5.2, повторился в 0.8.0 со статусными папками). Правила:
- **look**: в Trakt это строго show-only статус. Тип карточки вообще не проверяем — шлём S01E01 безусловно. Если пользователь каким-то образом положил в look фильм, Trakt просто ничего не сделает с seasons, на синхронизации карточка уедет куда положено.
- **viewed**: резолвим тип через `/search/tmdb/:id` без фильтра по типу, смотрим, что Trakt вернул. Если и show, и movie — приоритет show (в статусных папках Lampa почти всегда сериалы).

## Триггеры синхронизации

- **При открытии экрана Избранного** — полный проход: читаем watchlist + прогрессы, пересчитываем book/look/viewed/continued.
- Ручных кнопок синхронизации не нужно (thrown не реализуется).

## Зависимости и границы с `trakt_by_lampame`

Плагин `trakt_by_lampame` остаётся зависимостью, но используется **ограниченно**:

**Берём из него:**
- OAuth: авторизация, хранение токена в `Lampa.Storage.get('trakt_token')`, обновление по refresh-токену.
- Scrobbling эпизодов во время воспроизведения.
- UI авторизации в настройках (кнопка «Войти в Trakt» и т.п.).

**НЕ берём из него:**
- `window.TraktTV.api.upnext()` — использует VIP-эндпоинт `/sync/progress/up_next_nitro`, отдаёт некорректные данные. Вместо этого сами вызываем публичные эндпоинты `/sync/watched/shows` и/или `/users/me/progress/watched/activity?hide_completed=true`.
- В перспективе — и остальные API-методы (`watchlist`, `addToWatchlist` и т.п.) делаем сами. Это убирает зависимость от интерпретации данных чужим плагином.

**Базовый URL для наших запросов:** `https://apx.lme.isroot.in/trakt` — тот же прокси, который использует `trakt_by_lampame`. Прокси тупой: принимает Bearer-токен из `Lampa.Storage.get('trakt_token')`, на сервере подставляет `trakt-api-key` (client_id, под которым был выпущен токен), форвардит на `api.trakt.tv`. Уйти на прямой `api.trakt.tv` можно будет, только если зарегистрировать собственное приложение в Trakt и переиздать токен под его client_id — пока смысла не имеет. На интерпретацию ответов это не влияет: парсим Trakt-формат сами.

Канонические заголовки (мы тоже их используем):
- `Content-Type: application/json`
- `trakt-api-version: 2`
- `Authorization: Bearer <trakt_token>` (для авторизованных эндпоинтов)

**Параллельная работа:** `trakt_by_lampame` продолжает подгружать данные для своих home-рядов («Trakt Up Next», «Trakt Watchlist» и т.п.) при открытии главного экрана Lampa. Наш плагин читает независимо при открытии Избранного. Пересечений в запросах на запись нет, rate limit Trakt (1000/5мин) запасает на порядок. Если home-ряды не нужны — отключаются в настройках главного экрана Lampa, OAuth при этом продолжает работать.

## Задержки Trakt API и механизм Pending Ops

Trakt API может отдавать устаревшие данные 5–15 минут после записи (интерфейс Trakt и API часто рассинхронизированы — проверено опытом прошлых версий).

**Почему это важный кейс, а не пограничный:** типичный сценарий использования — пользователь добавляет фильм в папку и сразу открывает Избранное (пункт меню всегда под рукой). Это запускает полную синхронизацию до того, как Trakt успел перелить запись. Без дополнительных мер фильм «пропадёт» из папки на 5–15 минут после добавления. UX ломается.

**Решение — буфер Pending Ops (обязателен с первой версии):**

Плагин хранит в `Lampa.Storage` список недавних write-операций. Каждая запись:

```
{ id: <tmdb_id>, type: 'movie' | 'show', folder: 'book' | 'viewed' | 'look' | 'continued', action: 'add' | 'remove', ts: <unix_seconds> }
```

После каждого успешного write-запроса к Trakt (ответ 2xx) добавляется соответствующая запись в этот список.

На каждой полной синхронизации Избранного:

1. Читаем состояние Trakt (watchlist, прогрессы и т.п.).
2. Строим представление папок Lampa по данным Trakt.
3. **Накладываем сверху pending ops, которым меньше 15 минут:**
   - `action: 'add'` и Trakt ещё не показывает элемент → форсим наличие в папке.
   - `action: 'remove'` и Trakt ещё показывает → форсим отсутствие.
4. Pending ops старше 15 минут удаляем из списка (считаем, что Trakt уже догнал).

**Свойства механизма:**

- Trakt остаётся единственным долгосрочным источником правды. Pending ops — временный буфер, а не «локальное мнение» Lampa.
- Пользовательский UX не страдает от задержки API: сразу после действия Lampa показывает ожидаемое состояние.
- Идемпотентен: если Trakt догоняет до истечения окна, результат сверки не меняется (add+«Trakt уже показывает» = add без эффекта, и наоборот).

**Параметр окна:** 15 минут по умолчанию. Если окажется мало/много — вынесем в отладочные настройки.

## Что НЕ реализуется

Решено 2026-04-24, вынесено за скоуп:

- **Брошено (thrown)** — отложено до изучения логики «drop show» в Moviebase. Пока никак не мапится.
- **Нравится, Позже, Запланировано, История просмотров** — не трогаем.
- **Оценки/ratings** — не синхронизируются.
- **Поля `reset_at`, `hidden`** из progress — пока игнорируем. Базовую логику не меняют.

## Глоссарий

- **watchlist** (Trakt) — список «хочу посмотреть», без различия фильм/сериал.
- **up-next** (Trakt) — сериалы, у которых есть следующий непросмотренный эпизод.
- **progress** (Trakt) — объект с `aired`, `completed`, `next_episode` и т.п. для конкретного сериала.
- **scrobble** (Trakt) — событие start/pause/stop во время воспроизведения, которое автоматически помечает эпизод просмотренным.
