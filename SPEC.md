# SPEC — trakt_folder_sync

Спецификация поведения плагина. Пишется до кода. Язык — человеческий, не технический.

## Философия

Lampa — клиент Trakt. **Trakt — единственный источник правды.** Любое действие пользователя в Lampa транслируется в вызов Trakt API; Lampa никогда не хранит собственного состояния, которое может разойтись с Trakt. На каждой синхронизации Lampa зеркалит то, что сейчас в Trakt.

Последствие: петель «Lampa думает одно, Trakt другое» в принципе не возникает. Если Trakt отдал состояние — Lampa его показала, и точка.

## Категории Lampa

В Lampa два типа папок в Избранном:

- **Теги** — несколько одновременно: Закладки (book), Нравится, Позже, История просмотров.
- **Статусы** — только один: Смотрю (look), Просмотрено (viewed), Запланировано, Продолжение следует (continued), Брошено (thrown).

Плагин работает с: **book, look, viewed, continued, thrown**.

## Маппинг Trakt → Lampa

| Lampa | Trakt | Направление |
|---|---|---|
| Закладки (book) | Watchlist (фильмы + сериалы) | Двусторонняя |
| Смотрю (look) | Сериалы с `completed > 0 && aired > completed` | Trakt → Lampa |
| Просмотрено (viewed) | Фильм в истории / сериал полностью досмотрен и закрыт | Trakt → Lampa (производная) |
| Продолжение следует (continued) | Сериал догнан, но ещё выходит | Trakt → Lampa (производная) |
| Брошено (thrown) | Union трёх ячеек: `/users/hidden/progress_watched` ∪ `/users/hidden/dropped` ∪ выбранный custom list | Двусторонняя (read-union, тройной write) |

## Логика статусов сериала

Берём из Trakt:
- `show.status` (поле объекта шоу при `extended=full`) — одно из: `returning series`, `in production`, `planned`, `canceled`, `ended`.
- `progress.next_episode` — следующая серия, которую пользователю осталось посмотреть. Если `null` — пользователь догнан (это ровно тот же сигнал, по которому Trakt UI кладёт / не кладёт сериал в раздел Continue Watching).
- `progress.completed` — количество просмотренных (используется только чтобы отличить «смотрел хоть что-то» от «не смотрел ничего»).

**Решающий сигнал — `progress.next_episode == null`** («догнан»), а НЕ сравнение количества вышло/просмотрено и НЕ проверка `last_episode`.

Почему не `last_episode`: в эндпоинте `/shows/:id/progress/watched` поле `last_episode` — это последняя **просмотренная пользователем** серия, а не последняя вышедшая. Проверять её «просмотренность» бессмысленно — она по определению просмотрена.

Почему не сравнение количеств: если пользователь пропустил серию в середине, но добрал последнюю вышедшую, `aired > completed`, но смотреть прямо сейчас нечего. Trakt это понимает сам: `next_episode` в таком случае `null`.

Правила:

```
completed == 0
  → сериал не смотрели. В look/viewed/continued не попадает.
  → если в watchlist — остаётся только в Закладках.

completed > 0 && next_episode != null
  → есть что смотреть прямо сейчас → Смотрю.

completed > 0 && next_episode == null && status ∈ {ended, canceled}
  → сериал завершён и догнан → Просмотрено.

completed > 0 && next_episode == null && status ∈ {returning series, in production, planned}
  → пользователь догнал, ждёт новые серии → Продолжение следует.
```

Важно: «completed > 0» означает, что **хотя бы один эпизод отмечен в Trakt как просмотренный**. В Trakt это происходит при scrobble'е при воспроизведении либо при ручной отметке через API. Простое открытие карточки в Lampa это не триггерит.

Источник данных: `/sync/watched/shows?extended=full` даёт список сериалов с `show.status`; на каждый — отдельный запрос `/shows/:trakt_id/progress/watched` за `next_episode`. По сути это бесплатная реализация функциональности VIP-эндпоинта `up_next_nitro`: там Trakt отдаёт весь список Continue Watching разом, мы получаем тот же результат через 1+N запросов. Trakt rate limit 1000/5мин выдерживает десятки запросов на синхронизацию с запасом.

## Логика статусов фильма

Фильм попадает в **Просмотрено**, если он есть в истории Trakt (`/sync/history`, либо `plays > 0` у фильма в watched-списке).

Фильм в **Смотрю/Продолжение следует** не попадает никогда — эти статусы только для сериалов.

## Логика Брошено

Сценарий использования: пользователь работает с **несколькими клиентами одновременно** — несколько инстансов Lampa, Moviebase, веб-интерфейс Trakt. Все клиенты читают и пишут в Trakt; «брошенный» статус сериала должен быть виден во всех клиентах после действия в любом из них.

### Почему не одна ячейка Trakt

В Trakt нет нативного унифицированного статуса «Брошено». Вместо этого есть три независимые структуры, в каждую из которых пишет какой-то клиент, но никто не пишет одновременно во все три:

- **`/users/hidden/progress_watched`** (далее `H_PW`) — сериал скрыт из «Continue Watching». Это **то API, которое использует Moviebase** для действия «Stop watching» (подтверждено диагностикой 0.9.4-diag, 2026-04-26: после нажатия Stop watching в MB на Breaking Bad количество элементов в `H_PW` увеличилось ровно на 1, остальные ячейки не изменились).
- **`/users/hidden/dropped`** (далее `H_DR`) — старое API «Drop show». Это **то API, которое использует Trakt-веб** для действия «Drop show» (подтверждено той же диагностикой: после Drop в Trakt-вебе на Black Mirror количество элементов в `H_DR` увеличилось на 1, в `H_PW` не изменилось).
- **Custom list** пользователя в Trakt — обычный кастомный список, в который можно складывать сериалы. Сами по себе hidden-ячейки **скрывают** сериал из основных списков MB и Trakt-веба. Если положить только туда — сериал визуально «исчезнет» из этих UI. Custom list решает проблему: сериал появляется в навигации Trakt-веба и MB как отдельный список «Брошено», на постере появляется значок принадлежности к списку, и пользователь может в любой момент открыть список и увидеть свои брошенные сериалы. Hidden — это «канон-логика» MB, custom list — это «UX-метка».

Hidden-ячейки и custom list **не зеркалят друг друга автоматически**. Это поведение Trakt API, которое плагин обойти не может.

### Стратегия плагина: тройной write + read-union + write-канонизация

**Write при помещении сериала в Брошено** — параллельный POST в три ячейки одновременно:

```
POST /users/hidden/progress_watched   { shows: [{ ids: { tmdb } }] }
POST /users/hidden/dropped            { shows: [{ ids: { tmdb } }] }
POST /users/me/lists/<list_id>/items  { shows: [{ ids: { tmdb } }] }
```

Идемпотентно: повторный POST безвреден, Trakt не дублирует записи.

**Write при снятии с Брошено** — зеркально, в три ячейки:

```
POST /users/hidden/progress_watched/remove
POST /users/hidden/dropped/remove
POST /users/me/lists/<list_id>/items/remove
```

**Read при reconcile** — параллельный GET всех трёх ячеек, union по tmdb id. Сериал считается thrown, если найден хотя бы в одной.

**Write-канонизация при reconcile.** Если read-union показал `thrown`, но какие-то из трёх ячеек не содержат сериал (например, MB положил в `H_PW`, но `H_DR` и custom list пусты) — плагин **сам дописывает недостающие ячейки**. Так Lampa выступает «мостом» между клиентами, которые сами по себе не зеркалят: после открытия Лампы любой Lampa-юзер косвенно подтягивает совместимость для всех остальных клиентов одного аккаунта.

### Приоритет thrown в классификаторе

Hidden-ячейки **не сбрасывают** watched-историю в Trakt. Это значит, что брошенный сериал по-прежнему присутствует в `/sync/watched/shows` с тем же набором просмотренных эпизодов и тем же `progress.next_episode`. Если бы классификатор этого не знал — сериал тут же был бы переклассифицирован в look/viewed/continued по фактическому прогрессу.

Поэтому проверка thrown идёт **до** классификации look/viewed/continued: если сериал в read-union трёх ячеек — он thrown, дальше не идём. Если нет — обычная классификация по `next_episode`.

### Custom list — настройка пользователя, не автоматика

Плагин не создаёт custom list автоматически. Юзер заводит список в Trakt-вебе сам, плагин в настройках показывает селектор «Папка Trakt для статуса Брошено» с подгруженным через `GET /users/me/lists` перечнем custom-списков юзера. Юзер выбирает.

В настройках плагина хранится **`ids.trakt`** выбранного списка (числовой стабильный идентификатор), не `slug`. Slug ломается при переименовании списка, trakt id переживает переименование.

Если в настройках list не выбран — плагин работает в **деградированном режиме**: пишет только в `H_PW` + `H_DR` (двойной write), сериал считается thrown, но в MB/Trakt-вебе не будет визуального значка «список». Плагин в этом режиме поднимает warning в логе и поощряет настроить list.

При смене Trakt-аккаунта (смена токена) выбор сбрасывается, селектор пересобирается заново.

### Что Lampa не делает в логике Брошено

Не сбрасывает watched-историю при бросании. Hidden-ячейки и custom list сами не трогают `/sync/history` / `/sync/watched/shows`, и плагин этого тоже не делает. Прогресс просмотра пережил — пережил.

Не зеркалит MB ↔ Trakt-веб напрямую (нет Lampa-инстанса между ними). Если оба клиента работают параллельно без Lampa — каждый видит только свою ячейку. Это известное ограничение Trakt API, которое плагин в обход не лезет. Достаточно одного открытия Лампы, чтобы канонизация догнала состояние всех трёх ячеек.

### Pending Ops для thrown

Те же правила, что для look/viewed/continued (см. §«Задержки Trakt API…»). Pending op `add → thrown` форсит наличие в Брошено пока Trakt переливает запись (5–15 минут). Pending op `remove → thrown` форсит отсутствие. Без этого карточка «дёргается» обратно в thrown / из thrown пока reconcile тянет ещё устаревшие данные.

### Маппинг клиентов

| Клиент | Что пишет в Trakt при «бросить» | Что читает для «понять, брошен ли» |
|---|---|---|
| Lampa (наш плагин) | `H_PW` + `H_DR` + custom list (тройной) | union(`H_PW`, `H_DR`, custom list) |
| Moviebase | `H_PW` (только) | `H_PW` (только) |
| Trakt-веб | `H_DR` (только) | `H_DR` (только) |

Lampa — «расширенный клиент»: пишет шире всех, читает шире всех, дотягивает остальные клиенты до согласованного состояния. MB и Trakt-веб остаются «узкими» — их UX мы не меняем.

## Ручные действия пользователя

Принцип: **действие пользователя в Lampa = вызов Trakt API**. Lampa свой статус не меняет до подтверждения от Trakt (оптимистично отображает, но источник истины — Trakt после следующего чтения).

| Действие в Lampa | Вызов Trakt API |
|---|---|
| Добавить в Закладки | `POST /sync/watchlist` |
| Убрать из Закладок | `POST /sync/watchlist/remove` |
| Фильм → Просмотрено | `POST /sync/history` (добавить в историю) |
| Сериал → Просмотрено | `POST /sync/history` на все вышедшие эпизоды (перечень берём из `progress.seasons[].episodes[]`, шлём одним запросом). После этого Trakt отдаст сериал либо как Просмотрено (если ended/canceled), либо как Продолжение следует (если ещё выходит). Пользователь увидит конечный статус после следующей синхронизации. |
| Убрать из Просмотрено — фильм | `POST /sync/history/remove` с `movies`. |
| Убрать из Просмотрено — сериал | `POST /sync/history/remove` на все вышедшие эпизоды (тот же список, что при добавлении). |
| Сериал → Смотрю | `POST /sync/history` на `S01E01` (минимальная «подсказка Trakt'у»: completed становится > 0, next_episode указывает на S01E02, классификатор даёт look). Остальные эпизоды не трогаем. |
| Убрать из Смотрю | `POST /sync/history/remove` на `S01E01` — зеркало add'а. Отменяет нашу же подсказку. Реальную пользовательскую историю за пределами S01E01 НЕ трогаем: если она есть, классификатор оставит сериал в look/viewed/continued по данным Trakt — это корректно. |
| Любое → Брошено | Тройной параллельный write (см. §«Логика Брошено»): `POST /users/hidden/progress_watched` + `POST /users/hidden/dropped` + `POST /users/me/lists/<id>/items`. Прогресс просмотра не трогаем. |
| Убрать из Брошено | Зеркальный тройной remove: `POST /users/hidden/progress_watched/remove` + `POST /users/hidden/dropped/remove` + `POST /users/me/lists/<id>/items/remove`. После Pending окна следующий reconcile отдаст сериал в look/viewed/continued по фактическому прогрессу — отдельной логики «куда вернуть» не нужно. |
| Фильм → Смотрю, любое → Продолжение следует | **No-op.** Фильм в Смотрю не попадает в принципе. Продолжение следует — автоматическая производная «догнан + выходит», её пользователь не ставит. |

Ключевое: пользователь не «переносит» сериал в Просмотрено — он создаёт **условия**, при которых Trakt отдаст правильный статус. Если сериал ещё выходит, он окажется в Продолжение следует, а не в Просмотрено — это корректно.

Pending Ops для статусных папок (см. §«Задержки Trakt API...»): после успешной записи кладём op `{folder, action, id, type}`. На ближайших синхронизациях (пока op не протух) карточка не будет «уехать» из целевой папки из-за устаревших данных Trakt. Семантика: `add` в папку X → принудительно ТОЛЬКО в X (чистим id из остальных статусных desired); `remove` → чистим id из всех статусных desired.

**Определение типа карточки (show/movie) — через Trakt, не по полям карточки.** В листенере `Favorite.listener` Lampa регулярно отдаёт «тощие» карточки без `method`/`card_type`/`media_type`/`number_of_seasons` — проверять их эвристикой бессмысленно, отрицательный ответ «нет признаков сериала» ≠ «это фильм» (такой же баг был с book в 0.5.2, повторился в 0.8.0 со статусными папками). Правила:
- **book**: резолвим тип через `resolveCardType(card)` — сначала по полям карточки (быстро, для нормального добавления через UI), для тощих случаев `/search/tmdb/:id`. Шлём в `/sync/watchlist` ТОЛЬКО соответствующий тип (`movies` или `shows`, не оба).
- **look**: в Trakt это строго show-only статус. Тип карточки вообще не проверяем — шлём S01E01 безусловно. Если пользователь каким-то образом положил в look фильм, Trakt просто ничего не сделает с seasons, на синхронизации карточка уедет куда положено.
- **viewed**: резолвим тип через `/search/tmdb/:id` без фильтра по типу, смотрим, что Trakt вернул. Если и show, и movie — приоритет show (в статусных папках Lampa почти всегда сериалы).

**Важно про TMDB id и watchlist (баг 0.5.x–0.8.3):** До 0.8.4 add в book слался сразу в обе корзины Trakt (`movies: [{tmdb}]` и `shows: [{tmdb}]`), потому что предполагалось, что id из чужого namespace уйдёт в not_found. Это неверно: TMDB держит movies и shows в раздельных namespace, но один и тот же числовой id очень часто занят и фильмом, и сериалом — это разные сущности, и Trakt спокойно добавляет обе. Внутри плагина dedup по tmdb id маскировал пару (видим 1 элемент, в Trakt лежит 2), и мусор сидел в watchlist бесконечно. С 0.8.4 тип резолвим до отправки и шлём только нужный массив.

**Взаимное исключение статусных папок (с 0.8.6):** look/viewed/continued — взаимоисключающие mark'и в Lampa.Favorite (термин «mark» из `trakt_by_lampame.js:6031`: `MARK_FAVORITE_TYPES = ['look','viewed','scheduled','continued','thrown']`). Lampa сама не следит, чтобы карточка была только в одной mark-папке, и `Favorite.add(viewed, card)` НЕ снимает предыдущий continued/look mark. Симптом 0.8.5 (Running Point): пользователь вручную помечает viewed, в continued mark остаётся, и сериал виден в обеих папках. Per-folder `Favorite.remove(folder, card)` фаерит listener, но между итерациями карточка возвращается (видно в логе — никаких `Favorite.add`-событий между двумя `remove`-event'ами). С 0.8.6 после `computeStatusFolders` и перед per-folder `syncStatusFolder`'ами зовём `enforceStatusExclusivity(desired)`: для каждого классифицированного id вызываем `Lampa.Favorite.check(card)` (тот же API, что `trakt_by_lampame` использует в `removeConflictingMarks`) и `Favorite.remove` с любых конфликтующих status-папок, кроме той, куда классификатор хочет. После Promise.all — финальный `verifyStatusFoldersClean(desired)` перепроверяет результат и при необходимости повторяет remove. Это подменяет неработающий стандартный путь (per-folder remove → следующая итерация снова видит карточку в той же папке) на принудительный, тот же что использует штатный плагин.

**Reconcile при чтении watchlist (с 0.8.5):** при каждом чтении watchlist для каждого id, который есть и в Trakt, и в local Lampa book, проверяем тип через `resolveCardType(localCard)`. Если в watchlist лежит неверный тип — удаляем его. Если правильного типа в watchlist нет — добавляем. Это покрывает: (1) исторические пары movie+show с одним id; (2) случай «в watchlist остался только неправильный тип» (артефакт ошибочного cleanup'а 0.8.4 — там тип определялся через `cardIsShow`, на тощих карточках это давало ложный negative и удалялся не тот тип). С 0.8.5 определение типа идёт через `resolveCardType` (поля карточки → `/search/tmdb/:id` для тощих).

## Триггеры синхронизации

- **При открытии экрана Избранного** — полный проход: читаем watchlist + прогрессы, пересчитываем book/look/viewed/continued.
- Ручных кнопок синхронизации не нужно (thrown не реализуется).

## Зависимости и границы с `trakt_by_lampame`

Плагин `trakt_by_lampame` остаётся зависимостью, но используется **ограниченно**:

**Берём из него:**
- OAuth: авторизация, хранение токена в `Lampa.Storage.get('trakt_token')`, обновление по refresh-токену.
- Scrobbling эпизодов во время воспроизведения.
- UI авторизации в настройках (кнопка «Войти в Trakt» и т.п.).

**НЕ берём из него:**
- `window.TraktTV.api.upnext()` — использует VIP-эндпоинт `/sync/progress/up_next_nitro`, отдаёт некорректные данные. Вместо этого сами вызываем публичные эндпоинты `/sync/watched/shows` и/или `/users/me/progress/watched/activity?hide_completed=true`.
- В перспективе — и остальные API-методы (`watchlist`, `addToWatchlist` и т.п.) делаем сами. Это убирает зависимость от интерпретации данных чужим плагином.

**Базовый URL для наших запросов:** `https://apx.lme.isroot.in/trakt` — тот же прокси, который использует `trakt_by_lampame`. Прокси тупой: принимает Bearer-токен из `Lampa.Storage.get('trakt_token')`, на сервере подставляет `trakt-api-key` (client_id, под которым был выпущен токен), форвардит на `api.trakt.tv`. Уйти на прямой `api.trakt.tv` можно будет, только если зарегистрировать собственное приложение в Trakt и переиздать токен под его client_id — пока смысла не имеет. На интерпретацию ответов это не влияет: парсим Trakt-формат сами.

Канонические заголовки (мы тоже их используем):
- `Content-Type: application/json`
- `trakt-api-version: 2`
- `Authorization: Bearer <trakt_token>` (для авторизованных эндпоинтов)

**Параллельная работа:** `trakt_by_lampame` продолжает подгружать данные для своих home-рядов («Trakt Up Next», «Trakt Watchlist» и т.п.) при открытии главного экрана Lampa. Наш плагин читает независимо при открытии Избранного. Пересечений в запросах на запись нет, rate limit Trakt (1000/5мин) запасает на порядок. Если home-ряды не нужны — отключаются в настройках главного экрана Lampa, OAuth при этом продолжает работать.

## Задержки Trakt API и механизм Pending Ops

Trakt API может отдавать устаревшие данные 5–15 минут после записи (интерфейс Trakt и API часто рассинхронизированы — проверено опытом прошлых версий).

**Почему это важный кейс, а не пограничный:** типичный сценарий использования — пользователь добавляет фильм в папку и сразу открывает Избранное (пункт меню всегда под рукой). Это запускает полную синхронизацию до того, как Trakt успел перелить запись. Без дополнительных мер фильм «пропадёт» из папки на 5–15 минут после добавления. UX ломается.

**Решение — буфер Pending Ops (обязателен с первой версии):**

Плагин хранит в `Lampa.Storage` список недавних write-операций. Каждая запись:

```
{ id: <tmdb_id>, type: 'movie' | 'show', folder: 'book' | 'viewed' | 'look' | 'continued', action: 'add' | 'remove', ts: <unix_seconds> }
```

После каждого успешного write-запроса к Trakt (ответ 2xx) добавляется соответствующая запись в этот список.

На каждой полной синхронизации Избранного:

1. Читаем состояние Trakt (watchlist, прогрессы и т.п.).
2. Строим представление папок Lampa по данным Trakt.
3. **Накладываем сверху pending ops, которым меньше 15 минут:**
   - `action: 'add'` и Trakt ещё не показывает элемент → форсим наличие в папке.
   - `action: 'remove'` и Trakt ещё показывает → форсим отсутствие.
4. Pending ops старше 15 минут удаляем из списка (считаем, что Trakt уже догнал).

**Свойства механизма:**

- Trakt остаётся единственным долгосрочным источником правды. Pending ops — временный буфер, а не «локальное мнение» Lampa.
- Пользовательский UX не страдает от задержки API: сразу после действия Lampa показывает ожидаемое состояние.
- Идемпотентен: если Trakt догоняет до истечения окна, результат сверки не меняется (add+«Trakt уже показывает» = add без эффекта, и наоборот).

**Параметр окна:** 15 минут по умолчанию. С 0.8.3 значение настраивается в разделе «Trakt» → «Окно Pending Ops»: 15 мин / 5 мин / 1 мин / 10 сек / Выключено. «Выключено» полностью обходит буфер: ничего не сохраняется, на diff ничего не накладывается, синхронизация работает «как есть». Полезно для отладки, когда Trakt API быстро отдаёт свежие данные и ждать защитное окно незачем.

## Что НЕ реализуется

Вынесено за скоуп:

- **Нравится, Позже, Запланировано, История просмотров** — не трогаем.
- **Оценки/ratings** — не синхронизируются.
- **Поля `reset_at`** из progress — пока игнорируем. Базовую логику не меняют.

## Архитектура v2: серверный воркер (заметка на будущее)

Текущая реализация — чисто клиентский плагин на каждом устройстве. Для **multi-device** сценария (3–4 Лампы + 3–4 MB на одном Trakt-аккаунте) разумной эволюцией становится вынос синхронной логики на собственный backend.

Описаны **два варианта** — A и B. **B предпочтительнее** как финальная архитектура; A проще как промежуточный шаг. См. сравнение в конце раздела.

### Вариант A: серверный воркер только для канонизации и кеша (минимальный)

### Что серверный воркер решает

1. **Канонизация без UI-триггера.** Сейчас reconcile (включая канонизацию Брошено) случается только при открытии «Избранного» в Лампе. Cron-воркер раз в 5–10 мин читает три ячейки thrown, считает union, дописывает недостающее — независимо от того, открывал ли пользователь Лампу.
2. **Единый Pending Ops.** Сейчас Pending Ops хранится в `localStorage` каждого устройства независимо. Положил в Брошено на Лампе-TV1 — Лампа-TV2 в это же stale-окно (5–15 мин) увидит Trakt-state без нашего write и временно вернёт карточку в look. Сервер с общим буфером pending op'ов закрывает рассинхрон между устройствами.
3. **Разгрузка устройства.** ~9 GET'ов + N запросов на progress сериалов + классификация на TV-CPU ощутимы. Если сервер уже всё посчитал и отдаёт `desired-state` одним JSON'ом — плагин на устройстве становится тонкой синхронизацией локального `Lampa.Favorite` с этим JSON'ом. Быстрее на порядок.
4. **Кеш read-эндпоинтов Trakt.** При 4+ клиентах × N запросов есть риск 429-х; кеш на сервере снимает.
5. **Резолв конфликтов last-write-wins.** Когда два устройства одновременно меняют один и тот же tmdb id, сервер использует серверные timestamps (`watched_at`, `hidden_at`, `listed_at`) для выбора победителя. В однопользовательской клиентской схеме конфликта нет, в multi-device — становится регулярным.

### Чего серверный воркер НЕ решает

- **Stale-окно Trakt.** Race на стороне самого Trakt между write-API и read-API. Лечится тем же буфером pending op'ов — просто живёт он теперь на сервере.
- **Write-путь в UI Лампы.** Перетаскивание карточки в Брошено всё равно случается на устройстве, в `Lampa.Favorite.listener`. См. ниже про варианты (a) и (b).
- **localStorage Lampa.** На каждом устройстве свой. Сервер его не видит и не пишет туда напрямую. Плагин обязан запускаться на устройстве и синхронизировать локальный Favorite с присланным desired-state.
- **Внутренние особенности фронтенда Lampa** (двойные эмиты `Favorite.listener`, тонкие карточки, `Timeline.update` на скрабблинге). Это всё клиент.

### Возможная архитектура

```
   [MB×N]      [Lampa×N]
      \         /
       v       v
    [api.trakt.tv]   ← все клиенты пишут/читают как сейчас
        ^
        | каждые ~5 мин
        |
   [VPS-воркер: PHP/node/python — без разницы]
        |
        v
   [SQLite/Postgres: pending_ops, desired_state_cache, refresh_token]
```

### Endpoint'ы backend'а (минимум)

- `GET /api/desired-state?user_id=X` → готовый JSON `{look:[], viewed:[], continued:[], thrown:[], book:[]}` плюс episode-progress per show. Лампа-плагин синхронит локальный Favorite одним проходом.
- `POST /api/pending-op` → клиент сообщает «только что положил tmdb=X в folder=Y». Сервер кладёт в общий буфер на TTL=15 мин и применяет к desired-state, который отдаёт остальным клиентам.

### Cron-воркер (что делает регулярно)

- Тянет три ячейки thrown (`/users/hidden/progress_watched`, `/users/hidden/dropped`, `/users/me/lists/:id/items`), считает union, дописывает недостающее.
- Тянет `/users/me/watchlist/movies`, `/users/me/watchlist/shows`, `/sync/watched/movies`, `/sync/watched/shows?extended=full` — формирует кеш для read-стороны.
- Для каждого сериала с `progress_completed > 0` — `/shows/:trakt_id/progress/watched` для классификатора (look / viewed / continued).
- Чистит протухшие pending_ops (TTL).

### Дизайн write-пути: (a) vs (b)

- **(a) Lampa-плагин пишет напрямую в Trakt (как сейчас), сервер только читает и канонизирует.** Плюсы: клиент не зависит от доступности backend'а, действие летит в Trakt мгновенно. Минусы: дублирование кода (плагин знает Trakt-API, сервер тоже).
- **(b) Lampa-плагин шлёт действие на наш backend, сервер уже летит в Trakt.** Плюсы: минимальный код в плагине, единая точка контроля и логирования. Минусы: упал VPS — write не работает.

Рекомендация на старте — **(a)**: устойчивость к падениям VPS перевешивает; миграция от текущей архитектуры минимальная (плагин не меняет write-путь, добавляется только thin-read из `/api/desired-state` плюс `POST /api/pending-op`).

### Связь с задачей #8 (собственный OAuth)

Свой Trakt-прокси с собственным `client_id` — естественная часть VPS-инфраструктуры. Сейчас плагин ходит через общий прокси `apx.lme.isroot.in/trakt`. На VPS поднимается тонкий прокси (Cloudflare Worker, node-сервис, PHP — что угодно), который форвардит к `api.trakt.tv` и подставляет наш `client_id` из переменной окружения. Затем тот же воркер использует этот прокси для cron-задач, плагин — для write-пути.

### Когда стоит начинать

Не сейчас. Сначала добиваем текущий слой багов (book exclusivity, стабильность Брошено, episode sync) на чисто клиентской архитектуре. Серверный воркер не лечит логические баги — он переносит ту же логику на другое железо. Точка перехода — когда подключишь второй-третий клиент и упрёшься в рассинхрон между ними. К тому моменту клиентская логика должна быть стабильной, чтобы её можно было аккуратно перенести в серверный код.

### Вариант B: VPS как primary store (write-through cache + outbox)

В этой схеме VPS становится **единственным источником правды** для клиентов. Trakt из «авторитета» превращается в **destination** — пункт назначения, куда данные доползают в фоне. Клиенты с Trakt напрямую не разговаривают вообще.

#### Поток данных

```
   [Lampa×N]            [MB×N, Trakt-веб]
        \                   |
   POST  v                  v
   /action            api.trakt.tv
        |                   ^
        v          push     |   pull
   [VPS: БД + outbox + pull-loop] ──┘
        ^
   GET  |
   /state
        |
   [Lampa×N]
```

**Write-путь.** Lampa-listener → POST /action → VPS обновляет БД немедленно → outbox-воркер постит в Trakt с retry до 200/201.

**Read-путь.** Lampa открыла «Избранное» → GET /state → готовый JSON со всеми папками и episode-progress → синхронизация локального `Lampa.Favorite`.

**Pull от внешних клиентов.** VPS каждые 30–60 сек тянет Trakt и мержит с собственной БД. Это нужно, потому что MB и Trakt-веб пишут не нам, а напрямую в Trakt.

#### Что это решает

- **Stale-окно Trakt становится невидимым клиентам.** Race между Trakt write- и read-API теперь происходит между VPS и Trakt'ом — Lampa и MB-клиенты этого не видят. Pending Ops как клиентская концепция растворяются.
- **Мгновенная согласованность между Лампами.** Lampa-TV1 положила в Брошено → POST /action → VPS обновил БД → Lampa-TV2 при следующем GET /state видит уже свежий стейт. Нет 5–15 минут ожидания.
- **Robust write.** Сеть мигнула, Trakt вернул 5xx, rate limit — outbox-воркер ретраит. Клиент об этом не знает.
- **Тонкий клиент.** Плагин в Лампе — ~150 строк JS вместо текущих 2600. Только UI-listener'ы и локальная синхронизация Favorite с присланным state.

#### Тонкости реализации (НЕ забыть)

- **Idempotency-key.** Каждый POST /action клиент шлёт с `client_action_id` (uuid). VPS дедупит — на случай переотправки при потере связи.
- **Own-write-log.** VPS на 15 мин помнит, что только что запушил в Trakt. Pull-loop в этом окне игнорирует Trakt-state по этим tmdb id (иначе stale-Trakt откатит наш только что записанный action).
- **Conflict resolution last-write-wins.** При расхождении нашего state и pull-результата — побеждает свежий timestamp. У Trakt timestamps есть на всех важных эндпоинтах (`watched_at`, `hidden_at`, `listed_at`); у наших action'ов — `Date.now()` клиента или receipt-time на VPS.
- **Офлайн-fallback на клиенте.** VPS недоступен — плагин временно работает локально (пишет в `Lampa.Favorite` как сейчас + queue в localStorage), при восстановлении связи флашит queue в /action. Это аварийный путь, не основной.
- **Refresh-token хранится только на VPS.** Клиент логинится через VPS-OAuth-flow (callback на VPS), Lampa-плагин получает только session-cookie или короткий API-token для `/action`/`/state`. Это естественно сочетается с задачей #8.

#### Endpoint'ы backend'а (минимум)

- `GET /state` → `{look:[], viewed:[], continued:[], thrown:[], book:[], episodes:{tmdb:{seasons:[]}}}`. Версионируется (ETag/If-None-Match), чтобы Lampa могла дёргать часто и не платить трафиком.
- `POST /action` с `{client_action_id, type:'add'|'remove', folder, ids:{tmdb}, at}` → 200 «принято». Действие сразу применяется к state и кладётся в outbox.
- `POST /auth/callback` → завершает OAuth, кладёт refresh-token в БД.

#### Cron / loops на VPS

- **Outbox-воркер** (тикает по событию или каждые ~5 сек): читает первую необработанную запись из outbox, постит в Trakt, при успехе пометит выполненной, при 5xx/429 — backoff и ретрай.
- **Pull-loop из Trakt** (каждые 30–60 сек): тянет watchlist, watched_shows, watched_movies, hpw, hdr, list, прогресс по сериалам с completed > 0. Мержит с БД, применяя own-write-log и conflict resolver.
- **Канонизация Брошено** (каждые 5 мин): смотрит наш state, для каждой thrown-карточки проверяет, что она во всех трёх Trakt-ячейках; недостающие — кладёт в outbox.

### Сравнение A vs B

| Параметр | A (минимальный воркер) | B (primary store + outbox) |
|---|---|---|
| Где живёт source of truth | Trakt + локальный Lampa.Favorite | VPS |
| Write-путь Lampa → Trakt | напрямую через Trakt-API | через POST /action на VPS |
| Stale-окно Trakt видно клиентам | да, через клиентский Pending Ops | нет, скрыто на VPS |
| Согласованность между Лампами | через общий Pending Ops на сервере | мгновенная через /state |
| Размер клиента | ~1500 строк (текущие 2600 минус классификация и канонизация) | ~150 строк |
| Работа без VPS | да (плагин работает как сейчас) | только в офлайн-fallback'е |
| Сложность бэкенда | средняя (read + канонизация) | высокая (write + outbox + retry + idempotency + own-write-log + conflict resolver) |

A — пол-шага, дешёвый и устойчивый к падениям VPS. B — конечная цель, но требует больше серверной инженерии и делает клиентов завязанными на VPS.

Маршрут перехода, если когда-нибудь будем делать: сначала A (вынос канонизации и read-кеша на VPS), затем B (перенос write-пути через VPS-action API). Прыгать сразу в B можно, но потребует более длинного периода нестабильности при первой выкатке.

### Что в плагине останется после миграции на v2 (общее для A и B)

- Слушатели UI: `Favorite.listener`, `Timeline.listener` — клиентская природа, перенести нельзя.
- Локальная синхронизация `Lampa.Favorite` с desired-state, полученным от backend'а (через GET /state).
- В A — запись действий пользователя в Trakt напрямую. В B — запись действий через POST /action на наш backend.
- `_recentWrites` (защита от двойного эмита `Favorite.listener`) — клиентская особенность Lampa, остаётся.
- `_pendingLampaMark/Unmark` для эпизодов в stale-окне — в A остаётся, в B растворяется (stale-окно не видно клиенту).

Что **уйдёт** из плагина (в обоих вариантах): код классификации статусов, чтение Trakt из плагина, канонизация Брошено, кеш списков, episode-sync reconcile-логика. Всё это становится backend-задачей.

## Глоссарий

- **watchlist** (Trakt) — список «хочу посмотреть», без различия фильм/сериал.
- **up-next** (Trakt) — сериалы, у которых есть следующий непросмотренный эпизод.
- **progress** (Trakt) — объект с `aired`, `completed`, `next_episode` и т.п. для конкретного сериала.
- **scrobble** (Trakt) — событие start/pause/stop во время воспроизведения, которое автоматически помечает эпизод просмотренным.
- **canonization** (наш термин) — процесс «дописать недостающие ячейки» в трёх Trakt-target'ах для Брошено: если карточка есть в hpw + list, но нет в hdr — fire-and-forget POST в hdr.
- **stale-окно** (наш термин) — серверная задержка Trakt API между принятым write и его отражением на read-эндпоинтах того же объекта. Эмпирически 5–15 мин.
- **Pending Ops** (наш термин) — TTL-буфер недавних write-операций, который во время stale-окна Trakt подменяет недопроявившиеся изменения и не даёт reconcile «откатить» только что сделанное действие.
